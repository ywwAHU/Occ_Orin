# OCC数据采集方案

## 1. 目标

当前这套 `orin_occ` 工程已经收敛到 `V1` 方案：

- 主输入使用 `7` 个相机
- `3` 个前视
- `4` 个环视
- `5` 个激光雷达作为辅助输入，先转换成 `lidar_bev`

结合当前车辆配置，见 [大车.md](../当前环境/大车.md)，建议把数据采集目标明确成下面这句话：

**先采一套能支持 7 相机 OCC 训练和验证的数据，而不是一开始就把所有传感器、所有场景、所有标签问题一次做完。**

这件事很重要，因为 OCC 真正难的不是“录到数据”，而是：

- 时间同步是否足够稳定
- 标定是否可靠
- 190 度环视相机是否有可用的去畸变链路
- 后续能不能产出可训练的 occupancy 标签或伪标签

## 2. 当前建议的V1采集范围

### 必采

- `3` 个前视相机
- `4` 个环视相机
- `1` 个前向 `Robosense-M1P`
- `4` 个侧后向 `Robosense-E1R`
- `INS-5711`

### 建议采

- `ARS548` 毫米波

说明：

- 目前模型还没有直接吃毫米波分支，但采下来非常值钱，后面做融合或分析误检、漏检时会有帮助。
- 目前 `3` 个后向微光相机不进入 `V1` 主链路，所以**不要让这 3 路相机成为数据采集阻塞项**。

### 可选采

- `3` 个后向微光相机

建议：

- 如果录制系统带宽和存储压力不大，可以顺手录下来，单独归档。
- 如果录制系统压力很大，`V1` 可以先不录，先把 `7` 相机主链路做好。

## 3. 先做什么，不要先做什么

### 先做

- 先把 `7` 相机 + `5` 雷达 + `INS` 的同步采集跑通
- 先验证每个传感器时间戳是否稳定
- 先验证相机和雷达外参是否基本可用
- 先验证环视相机的去畸变结果是否可用于 OCC
- 先录一小批样本做链路联调

### 不要一上来就做

- 不要一开始就录几十上百小时
- 不要一开始就追求全量人工标注
- 不要先录成“只有视频，没有时间戳和外参”的离线资料
- 不要先把后三个微光相机也纳入主采集约束

## 4. 采集分三阶段

## 阶段A：链路打通

目标：

- 验证录制程序稳定
- 验证同步关系
- 验证标定文件齐全
- 验证可以生成训练 manifest

建议规模：

- `30` 分钟到 `1` 小时

建议场景：

- 白天城市道路
- 白天园区道路
- 停车场低速绕行
- 路口转弯

这一阶段验收标准：

- 所有传感器都有时间戳
- 时间戳单调递增，没有大段跳变
- 相机和雷达能按同一时刻对齐
- `build_vehicle_lidar_manifest.py` 能跑通
- 能抽取若干帧进入当前 `orin_occ` 数据格式

## 阶段B：小规模试采

目标：

- 形成第一批可用于开发的原始数据
- 建立数据清洗和抽帧流程
- 验证伪标签或标注方案是否可行

建议规模：

- `5` 到 `10` 小时原始录制

建议场景分布：

- 白天正常路况
- 傍晚逆光
- 夜间有路灯
- 雨天或湿地路面
- 窄路会车
- 大车遮挡
- 行人和非机动车密集区域
- 停车场和低速绕障

这一阶段输出：

- 原始采集目录
- 标定包
- 质量检查报告
- 一批可以转成训练 manifest 的样本

## 阶段C：正式采集

目标：

- 为 OCC 训练准备主数据集

建议规模：

- `20` 到 `50` 小时作为第一版目标

说明：

- 先做到 `20` 小时高质量数据，比匆忙录 `100` 小时低质量数据更有价值。
- 真实可用时长通常会低于原始录制时长，因为会剔除同步异常、标定异常、严重过曝、严重雨污遮挡和传感器缺失片段。

## 5. 每条数据必须包含什么

每个采集片段必须至少有下面这些内容：

- 传感器原始数据
- 每帧或每包时间戳
- 标定文件
- 车辆基本状态
- 采集环境描述

### 相机侧

建议保存：

- 原始图像或高质量编码视频
- 每帧精确时间戳
- 相机内参
- 畸变参数
- 相机到车体坐标系外参
- 分辨率和曝光相关元数据

建议：

- `190` 度环视相机一定要保留原始鱼眼模型参数，不要只留去畸变后的图片。
- 如果后处理会生成去畸变图，也要保留“原图 -> 去畸变图”的映射关系。

### 激光雷达侧

建议保存：

- 每个雷达的原始点云
- 每帧时间戳
- 雷达到车体坐标系外参
- 点云格式说明

建议命名：

- `front_m1p`
- `front_e1r`
- `rear_e1r`
- `left_e1r`
- `right_e1r`

说明：

- 当前工程已经支持把多雷达原始数据转成 `lidar_bev`。
- 但前提是每个雷达都有可靠的 `lidar_to_ego` 或 `ego_to_lidar` 外参。

### INS侧

建议保存：

- 时间戳
- 经纬度
- 姿态角或四元数
- 速度
- 加速度
- 角速度

说明：

- 即便当前模型还没有时序分支，`INS` 也是后续做点云累积、伪标签生成、时序 OCC 的关键。

### 毫米波侧

建议保存：

- 原始目标列表或点迹
- 每帧时间戳
- 雷达到车体坐标系外参

说明：

- `V1` 不是必需项，但如果采集系统顺手能带上，强烈建议保留。

## 6. 时间同步要求

如果现在只能定一个最重要的技术要求，那就是：

**所有传感器必须有统一、可靠、可追溯的时间戳。**

建议目标：

- 相机之间时间偏差尽量控制在 `10 ms` 以内
- 相机与雷达对齐误差尽量控制在 `20 ms` 以内
- `INS` 时间戳要稳定并可映射到相机和雷达时间轴

如果现阶段做不到严格硬同步，至少要做到：

- 全部传感器使用同一时间基准
- 每帧时间戳来源清楚
- 后处理时可以按时间戳进行最近邻或插值对齐

## 7. 标定要求

每次正式采集都要绑定一份标定包，至少包含：

- `7` 个相机内参
- `7` 个相机畸变参数
- `7` 个相机外参
- `5` 个雷达外参
- `INS` 到车体坐标系变换

建议：

- 每次传感器支架、角度、安装位置有变化，就重新标定
- 每天采集开始前做一次快速校验
- 每周或每轮任务做一次完整复核

## 8. 数据目录建议

建议每次采集按 `trip` 组织，而不是把所有文件混在一起。

推荐目录：

```text
raw_data/
  trip_20260415_day_city_001/
    cameras/
      front_left/
      front_center/
      front_right/
      surround_left_front/
      surround_right_front/
      surround_left_rear/
      surround_right_rear/
    lidars/
      front_m1p/
      front_e1r/
      rear_e1r/
      left_e1r/
      right_e1r/
    ins/
    radar/
    calib/
    meta/
```

`meta/` 里建议放：

- 场景说明
- 采集日期
- 采集人员
- 天气
- 时间段
- 设备版本
- 标定版本
- 异常记录

## 9. 当前工程需要的原始manifest格式

为了接入当前 `orin_occ`，建议把原始采集数据先整理成一份 `JSONL manifest`。

仓库里已经有一个示例文件：

- [vehicle_raw_manifest.example.jsonl](../orin_occ/examples/vehicle_raw_manifest.example.jsonl)

这个原始 manifest 需要至少包含：

- `sample_id`
- `cameras`
- `lidars`
- 可选的 `occupancy`
- 可选的 `timestamp`
- 可选的 `metadata`

之后可以用下面这个脚本转成当前训练直接可用的 manifest：

```bash
PYTHONPATH=src python scripts/build_vehicle_lidar_manifest.py \
  --config configs/baseline_orin_camera_lidar.json \
  --input-manifest /path/to/vehicle_raw_train.jsonl \
  --output-manifest work_dirs/manifests/vehicle_train_7cam.jsonl \
  --bev-dir work_dirs/lidar_bev/train
```

## 10. 标签问题要提前想清楚

这是 OCC 数据采集里最容易被低估的地方。

### 事实

- 你们现在可以先采原始数据
- 但如果没有 occupancy 标签，就不能直接做标准监督训练

### 当前最现实的路线

第一步：

- 继续用公共数据做预训练

第二步：

- 用实车多雷达 + `INS` 累积生成伪标签

第三步：

- 对关键场景做人工抽检和修正

建议你们不要等“标注方案完全确定”后再开始采集，但要在采集前先确定下面两件事：

- 点云累积坐标系怎么定义
- 伪标签生成需要多长时间窗口

## 11. 建议优先采哪些场景

如果资源有限，优先级建议如下：

第一优先级：

- 路口转弯
- 贴边行驶
- 窄路会车
- 停车场低速绕障
- 大车遮挡

第二优先级：

- 夜间城市道路
- 傍晚逆光
- 雨天反光路面
- 行人和两轮车密集区域

第三优先级：

- 高速或快速路
- 极端天气

原因：

- 你们这版 OCC 更偏向近场和中近场占用建模，第一阶段更应该把复杂局部场景录扎实。

## 12. 质量验收清单

每次采集结束后，至少做下面这些自动或人工检查：

- 是否存在传感器缺包
- 是否存在长时间时间戳停滞
- 是否存在相机严重模糊或污渍遮挡
- 是否存在雷达点云全黑或空帧
- 标定版本是否匹配
- 场景标签是否填写完整
- 数据目录是否能自动生成 manifest

建议做一个简单日报，记录：

- 总录制时长
- 可用时长
- 问题片段
- 标定状态
- 设备异常

## 13. 存储建议

不要等磁盘爆了再回头补方案。

粗略建议：

- 先做一次 `10` 分钟试录
- 实测每小时数据量
- 再反推整轮采集所需容量

经验上，多相机、多雷达原始数据的体量通常会很快上来，所以建议：

- 原始数据单独存一份
- 中间处理结果单独存一份
- 训练抽样集再单独存一份

不要把原始数据和中间产物混在同一个目录。

## 14. 我对你们当前最实际的建议

如果现在就开工，我建议按下面顺序推进：

1. 先把 `7` 相机 + `5` 雷达 + `INS` 录制跑通
2. 做 `30` 分钟到 `1` 小时链路验证采集
3. 整理出第一版原始 `JSONL manifest`
4. 用当前仓库的脚本把多雷达转换成 `lidar_bev`
5. 抽取一小批样本，验证训练输入链路
6. 再决定正式采集规模

## 15. 当前结论

你们现在最应该做的不是“马上录很多数据”，而是：

**先把一条标准、稳定、可复用的数据采集链路建立起来。**

对当前这套 OCC 工程来说，一条合格的采集链路至少要满足：

- `7` 相机主链路完整
- `5` 雷达原始数据可回放
- `INS` 时间轴可靠
- 标定包完整
- 可以生成当前工程可读的 manifest

只要这五件事成立，后面的伪标签、训练和部署都会顺很多。

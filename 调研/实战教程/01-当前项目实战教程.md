# 当前项目实战教程

## 这篇文档解决什么问题

前面的文档已经把概念、硬件、数据、OCC 原理讲清楚了。

这篇文档不再讲“这个领域大概是什么”，而是直接回答：

**以你们当前这台车、当前这套仓库、当前这个阶段，接下来到底该怎么一步一步推进。**

你可以把它理解成：

- 一份面向当前现状的项目作战手册

## 一、先把当前现状说清楚

根据当前仓库里的最新信息，你们现在的真实情况大致是这样：

### 车辆硬件

- 域控：`Jetson AGX Orin 32G`
- 相机：`10` 路
- 前视：`3` 路
- 环视：`4` 路，均为 `190` 度
- 后向微光：`3` 路
- 激光雷达：`5` 个
- 前向：`Robosense-M1P`
- 前后左右：`4` 个 `Robosense-E1R`
- 毫米波：`ARS548`
- 组合导航：`INS-5711（含 RTK）`

### 当前工程主线

当前仓库 `orin_occ/` 已经收敛到一个更现实的 `V1` 路线：

- 主相机输入用 `7` 路
- 也就是：前视 `3` 路 + 环视 `4` 路
- 后向 `3` 路微光相机先不进入 `V1` 主链路
- `5` 个激光雷达先通过预处理变成 `lidar_bev`
- `INS` 必须保留
- 毫米波建议采集，但当前不作为主模型必需输入

### 当前代码状态

这套 baseline 目前已经具备下面这些能力：

- `7` 相机主配置已经跑通
- synthetic training 能跑通
- ONNX 导出能跑通
- 已支持公共 `Occ3D` 数据启动
- 已支持多雷达原始数据桥接成 `lidar_bev`

### 当前真正的主要矛盾

不是：

- 再找一个更新的论文
- 再换一个更重的模型

而是：

- 数据采集链路还没完全标准化
- 实车数据到 manifest 的流程还没形成固定规范
- 实车 occupancy 标签或伪标签方案还没定型
- `190` 度环视的几何处理是明显风险点

## 二、这篇教程的最终目标

你们当前阶段不应该追求“一步做到量产级系统”。

更现实的目标是：

**在你们这套车上，建立一个从试采数据到训练输入的最小可用闭环。**

这个闭环至少要包含：

1. 有一批试采原始数据
2. 有配套标定包和时间戳
3. 能整理出原始 manifest
4. 能把多雷达转成 `lidar_bev`
5. 能让当前 `orin_occ` 成功读到样本
6. 能做小规模训练或至少 loader 验证

只要这个闭环建立起来，项目就真正从“调研阶段”进入“可持续工程阶段”。

## 三、整个实战分成 5 个阶段

## 阶段 0：先把环境和主线定住

### 当前阶段目标

不是马上采数据，而是先统一团队认知。

### 你现在就要统一的结论

- `V1` 主相机只用 `7` 路
- `3` 路后向微光暂不纳入主链路
- `5` 雷达保留，但先通过 `lidar_bev` 桥接
- `INS` 必须录
- 毫米波尽量录，但不阻塞主链路

### 这一步的产出

- 所有人对“当前主线输入”没有歧义
- 数据采集同学知道哪些是必录，哪些是建议录，哪些不是当前阻塞项

### 这一步的验收标准

你能用一句话准确复述当前方案：

**我们先做 7 相机主干 + 5 雷达辅助 + INS 保留的 V1 OCC，不把后向 3 路微光作为当前主链路刚需。**

## 阶段 1：把 baseline 保持在“随时可验证”的状态

为什么要先做这一步？

因为如果仓库本身都不稳定，后面实车数据一进来，你就分不清问题到底来自：

- 数据
- 还是代码

### 当前推荐做法

先在训练服务器上确认下面三条命令随时能跑。

```bash
cd /root/autodl-tmp/orin_occ
PYTHONPATH=src /root/miniconda3/bin/python scripts/smoke_test.py --config configs/baseline_orin_camera_lidar.json
```

```bash
cd /root/autodl-tmp/orin_occ
PYTHONPATH=src /root/miniconda3/bin/python scripts/train.py --config configs/baseline_orin_camera_lidar.json
```

```bash
cd /root/autodl-tmp/orin_occ
PYTHONPATH=src /root/miniconda3/bin/python scripts/export_onnx.py --config configs/baseline_orin_camera_lidar.json
```

### 为什么这一步重要

它相当于在说：

- 当前 baseline 没坏
- 后面一旦实车数据接不进来，大概率是数据链路问题，而不是仓库本身先坏了

### 这一步的产出

- baseline 可运行确认
- 当前配置文件和脚本路径确认

### 这一步的验收标准

至少满足：

- smoke test 正常
- synthetic training 正常
- ONNX 导出正常

## 阶段 2：做一次“小而完整”的试采

这是当前最重要的实战阶段。

### 试采的目标

不是追求大量数据。

而是验证下面这条链是否成立：

- 传感器能稳定录
- 时间戳完整
- 标定文件齐
- 数据目录规范
- 后面能整理成 manifest

### 推荐试采规模

- `30` 分钟到 `1` 小时

### 推荐试采场景

- 白天园区道路
- 低速转弯
- 停车场绕行
- 近场会车

### 必录内容

- 前视 `3` 相机
- 环视 `4` 相机
- `5` 个激光雷达
- `INS`

### 建议录

- `ARS548`

### 当前不要让它阻塞主线的内容

- 后向 `3` 路微光相机

### 试采后的原始目录建议

```text
raw_data/
  trip_20260416_test_001/
    cameras/
      front_left/
      front_center/
      front_right/
      surround_left_front/
      surround_right_front/
      surround_left_rear/
      surround_right_rear/
    lidars/
      front_m1p/
      front_e1r/
      rear_e1r/
      left_e1r/
      right_e1r/
    ins/
    radar/
    calib/
    meta/
```

### `calib/` 里至少要有

- `7` 相机内参
- `7` 相机畸变参数
- `7` 相机外参
- `5` 雷达外参
- `INS` 到车体坐标系变换

### `meta/` 里建议写清楚

- trip 名称
- 时间段
- 天气
- 采集地点
- 标定版本
- 录制软件版本
- 备注异常

### 这一步的验收标准

这次试采结束后，你必须能回答：

- 每个传感器时间戳在哪
- 标定包在哪
- 哪个 trip 是这次试采
- 哪几路是 V1 主相机

如果你回答不清楚，说明试采还不算真正可用。

## 阶段 3：把试采原始数据整理成 manifest

这一步是“原始数据”走向“训练数据”的桥。

### 第一步：先做原始 manifest

格式参考仓库里的例子：

- `orin_occ/examples/vehicle_raw_manifest.example.jsonl`

一个原始样本至少要有：

- `sample_id`
- `cameras`
- `lidars`
- 可选的 `timestamp`
- 可选的 `metadata`
- 可选的 `occupancy`

### 一个简化样本大概长这样

```json
{
  "sample_id": "trip_20260416_test_001_000123",
  "timestamp": 1776206400123456,
  "cameras": {
    "front_left": {
      "image": "/data/raw_data/trip_20260416_test_001/cameras/front_left/1776206400123456.jpg",
      "intrinsic": [[...], [...], [...]],
      "ego_to_camera": [[...], [...], [...], [...]]
    }
  },
  "lidars": {
    "front_m1p": {
      "points": "/data/raw_data/trip_20260416_test_001/lidars/front_m1p/1776206400123456.npy",
      "lidar_to_ego": [[...], [...], [...], [...]]
    }
  },
  "metadata": {
    "trip_id": "trip_20260416_test_001"
  },
  "occupancy": ""
}
```

### 这里最容易出错的地方

- 相机名字和配置文件不一致
- 图像路径错
- 内参不是 resize 后对应值
- `ego_to_camera` 搞反
- `lidar_to_ego` 缺失或不可靠

### 当前 7 相机名字建议固定成

- `front_left`
- `front_center`
- `front_right`
- `surround_left_front`
- `surround_right_front`
- `surround_left_rear`
- `surround_right_rear`

### 当前 5 雷达名字建议固定成

- `front_m1p`
- `front_e1r`
- `rear_e1r`
- `left_e1r`
- `right_e1r`

### 这一步的验收标准

你应该能生成一份：

- `vehicle_raw_trial.jsonl`

并且其中至少有几十到几百个样本行，字段完整、路径可访问、命名一致。

## 阶段 4：把多雷达转成 `lidar_bev`

这一步是把原始多雷达数据接进当前 baseline 的关键。

当前仓库已经有现成脚本：

- `scripts/build_vehicle_lidar_manifest.py`

### 推荐命令

```bash
cd /root/autodl-tmp/orin_occ
PYTHONPATH=src /root/miniconda3/bin/python scripts/build_vehicle_lidar_manifest.py \
  --config configs/baseline_orin_camera_lidar.json \
  --input-manifest /path/to/vehicle_raw_trial.jsonl \
  --output-manifest work_dirs/manifests/vehicle_trial_7cam.jsonl \
  --bev-dir work_dirs/lidar_bev/trial
```

如果你的雷达原始 `.bin` 点云每个点包含 `5` 个 float，比如：

- `x y z intensity ring`

那就加：

```bash
  --bin-columns 5
```

### 这个脚本做了什么

它会：

1. 读取原始 manifest
2. 把每个雷达点云通过外参变到 ego 坐标系
3. 融合 `5` 个雷达点云
4. 栅格化成 `lidar_bev`
5. 写出当前训练直接能读的新 manifest

### 生成后的输出通常包括

- 一批 `.npy` 的 `lidar_bev`
- 一份 `vehicle_trial_7cam.jsonl`

### 这一步的验收标准

至少满足：

- 命令能跑通
- 成功生成新的 manifest
- manifest 里 `lidar_bev` 路径真实存在
- 生成的 `lidar_bev` 形状和当前配置一致

当前默认应该是：

- `4 x 128 x 128`

## 阶段 5：做“小样本读数验证”

这一步不要急着一上来全量训练。

先验证：

- 当前样本能不能被代码正常读取

### 最稳妥的做法

先把新 manifest 配到一个试验配置里，或者直接临时替换 `training.manifest_path`，然后跑：

- loader 检查
- 单步训练检查

### 一个现实的建议

你现在可以先做一个很小的 trial manifest：

- 比如 `50` 到 `200` 个样本

这样有几个好处：

- 出错更快暴露
- 排查成本更低
- 不会因为数据量大把日志淹没

### 当前你最应该验证的 5 件事

1. `7` 路图像是不是都能读到
2. 相机名字和顺序是否正确
3. `lidar_bev` 是否成功加载
4. `occupancy` 为空时流程是否符合你当前预期
5. 当前几何输入是否没有明显 shape 错误

### 这一步的验收标准

至少满足其中一个：

- 可以完成一次 loader 读样本
- 可以完成一次单步训练前向

## 四、当前项目最关键的现实判断

## 判断 1：现在先不要大规模实车训练

原因很简单：

- 你们还没有稳定的实车 occupancy 标签主供给

没有标签路线前，大规模实车训练很容易变成：

- 有很多原始数据
- 但训练闭环并不稳

## 判断 2：当前真正有积累价值的是“数据闭环”

当前最值钱的进展不是：

- 指标先多高

而是：

- 试采规范确定
- manifest 规范确定
- 多雷达桥接确定
- 7 相机主链路确定

这些一旦稳定，后面无论做伪标签、时序、鱼眼建模还是部署优化，都有了可靠底座。

## 判断 3：当前环视几何问题必须尽早面对

你们 4 个环视都是 `190` 度。

这意味着：

- 不能默认它们按普通针孔模型无痛接入

你们要尽快搞清楚：

- 当前训练用图像是原始鱼眼图，还是去畸变图
- 如果 resize 了，内参怎么同步更新
- 原图和去畸变图的映射关系有没有被保存

如果这件事不清楚，后面即使数据接进来了，空间结果也可能一直不稳。

## 六、常见问题先查什么

## 问题 1：manifest 能生成，但训练读不到图

先查：

- 图片路径是否真实存在
- 相机名字是否和配置一致
- 路径分隔符和平台路径是否一致

## 问题 2：图像能读，但空间结果明显怪

先查：

- 内参是否对应 resize 后图像
- `ego_to_camera` 是否方向反了
- 环视是不是直接拿原始鱼眼图硬上

## 问题 3：多雷达桥接结果几乎全空

先查：

- `lidar_to_ego` 是否正确
- 点云单位和坐标方向是否一致
- `bin_columns` 是否设置对
- 点云是否落在当前网格范围内

## 问题 4：样本能读，但训练不稳定

先查：

- 当前是不是还没有可靠 occupancy 标签
- 是否把“无标签原始数据”误当成“可监督训练数据”
- 相机和雷达时间是否真的对齐

## 七、这篇教程读完以后，你应该真的能做什么

不是“好像懂了”。

而是至少能自己推动下面这些事情：

- 说清楚当前 V1 输入到底是什么
- 指挥一次有标准的试采
- 要求团队把数据按固定目录落盘
- 整理出一份原始 manifest
- 运行多雷达桥接脚本
- 验证样本能否接入当前 baseline

只要你能独立推进这几件事，你就已经不是“只会看文档的人”了，而是在真正开始推进项目。

## 八、最后一句最重要的话

对你们当前现状来说，最正确的推进方式不是：

- 一头扎进更复杂模型

而是：

**先把 7 相机主链路、5 雷达桥接、试采规范、manifest 生成和小样本闭环这五件事做扎实。**

这五件事一旦站稳，后面的标签、伪标签、时序、部署优化才会真正有抓手。

# Orin 车端部署 OCC 方案

## 1. 背景

当前车端基础环境为 `Jetson AGX Orin 32GB`。目标是在车上部署 OCC（Occupancy）感知能力，并满足车端推理对实时性、稳定性和可维护性的要求。

需要先明确一点：在 Orin 上部署 OCC，通常不是把训练侧的 PyTorch 工程直接搬到车上运行，而是需要做一轮面向部署的模型重构、算子适配和 TensorRT 化。

---

## 2. 核心结论

在 Orin 上部署 OCC，推荐采用以下主路线：

- 推理主设备使用 `GPU`
- 推理框架使用 `TensorRT`
- 精度优先选择 `FP16`
- 模型结构优先选择“部署友好”的 `dense BEV` 路线

不推荐一开始就采用以下路线：

- 直接在车端运行完整 PyTorch 推理
- 一开始就做 `INT8` 量化
- 将整个 OCC 网络强行压成一个超大的 TensorRT engine
- 依赖大量 `3D sparse conv / spconv` 而不做结构改造
- 指望 OCC 主干运行在 `DLA`

原因是 Orin 虽然算力强，但 TensorRT 对标准 ONNX 算子更友好；一旦模型中包含大量稀疏卷积、体素散射、自定义 pooling、复杂 attention 等算子，部署难度会明显上升，很容易从“模型导出问题”变成“插件开发项目”。

---

## 3. 适合 Orin 的总体思路

### 3.1 推荐路线

最稳妥的思路是：

1. 选择部署友好的 OCC 模型结构
2. 将模型拆成若干可单独部署的模块
3. 将核心推理部分转成 TensorRT
4. 在 Orin 本机上构建 engine 并做性能验证
5. 最后再接入车端数据流和上下游模块

### 3.2 为什么主推 GPU + TensorRT + FP16

- `GPU` 是 Orin 上最现实、最灵活的主推理资源
- `TensorRT` 是 NVIDIA 官方主推的高性能推理框架
- `FP16` 往往可以在精度、性能和落地难度之间取得较好平衡
- 对 OCC 来说，第一阶段更重要的是“稳定跑通”，不是一上来追求极限吞吐

### 3.3 为什么不建议依赖 DLA

`DLA` 更适合规则、受限的小型 CNN 子图，不适合作为 OCC 主体部署平台。原因包括：

- 支持的层类型有限
- 对动态 shape 支持差
- 对网络结构限制多
- 不适合复杂的 OCC 图结构

因此，DLA 只能作为可选优化方向，不应作为主方案。

---

## 4. 模型选择建议

### 4.1 优先选择的模型类型

如果当前还没有完全锁定模型，建议优先考虑以下结构：

- `camera-only OCC` 中偏 `dense BEV` 的方案
- 将 3D 表达尽早转换成规则 BEV 特征图的方案
- 主要由标准卷积、上采样、拼接、激活组成的网络

这一类模型更容易完成：

- ONNX 导出
- TensorRT 构图
- FP16 部署
- 后续性能调优

### 4.2 不推荐直接照搬的模型类型

以下类型在 Orin 上部署难度通常较高：

- 大量使用 `spconv` 的 lidar OCC
- 严重依赖稀疏体素算子的模型
- 需要复杂自定义 scatter/gather 的模型
- 使用大量自定义 attention 或 CUDA-only 算子的模型

这些模型并不是完全不能部署，而是通常需要：

- 自定义 TensorRT plugin
- 单独编写 CUDA 算子
- 拆分引擎
- 大量数值对齐和性能调试

如果项目周期紧，建议优先选择部署友好的架构，而不是把所有难点压到部署阶段解决。

---

## 5. 适合车端的模块拆分方式

在车端落地时，不建议把整套 OCC 全部打包成一个大 engine。更建议按模块拆开：

### 5.1 推荐模块划分

1. 预处理模块
2. Backbone / Neck 模块
3. BEV 变换或 View Transform 模块
4. OCC Head 模块
5. 后处理与结果发布模块

### 5.2 各模块建议

#### 预处理模块

包含：

- 图像去畸变
- resize / crop
- normalize
- 相机参数处理
- 时间同步

建议使用：

- `C++`
- `CUDA`
- 或者 NVIDIA 多媒体/图像处理链路

#### Backbone / Neck

这部分通常最适合直接转 `TensorRT`，也是部署中最容易获得收益的部分。

#### View Transform / Voxel Pooling

这是 OCC 部署中的关键难点之一：

- 如果是标准算子可表达的版本，优先放入 TensorRT
- 如果 ONNX 无法稳定表达，建议单独做 CUDA 或 plugin
- 如果模块过于复杂，建议单独拆成一个运行模块，而不是强塞到主 engine

#### OCC Head

这部分通常可以继续使用 TensorRT，只要结构保持规整、输入输出 shape 清晰即可。

#### 后处理模块

包含：

- occupancy 阈值化
- 栅格重排
- 置信度过滤
- 发布给下游规控或可视化系统

建议放在 `C++/CUDA`，便于控制延迟和集成。

## 6. 推荐部署流程

### 6.1 第一步：冻结车端软件环境

在正式部署前，先固定以下版本：

- `JetPack`
- `CUDA`
- `cuDNN`
- `TensorRT`
- 驱动和系统镜像版本

这是第一优先级。因为 TensorRT engine、plugin 编译结果、运行时行为都和软件版本强相关。如果版本不固定，后续会出现大量重复排障工作。

### 6.2 第二步：做模型部署可行性体检

在导出模型之前，先分析网络结构，把每个模块归类为：

- 可以直接 ONNX 导出
- 可以被 TensorRT 原生支持
- 需要 graph rewrite
- 需要 plugin
- 需要改模型结构

建议最终形成一张部署清单，例如：

| 模块 | 处理方式 |
|------|----------|
| Backbone | TensorRT |
| Neck | TensorRT |
| View Transform | TensorRT / Plugin / CUDA |
| OCC Head | TensorRT |
| Postprocess | C++ / CUDA |

### 6.3 第三步：从训练侧导出 ONNX

建议策略如下：

- 固定输入分辨率
- 固定时序长度
- 固定 batch size 为 `1`
- 去掉训练侧专用逻辑
- 如果整图导出困难，就按模块拆分导出

第一阶段目标不是“一步导出全模型”，而是先导出最小可运行版本。

### 6.4 第四步：使用 TensorRT 做基线验证

建议先做 `FP16` 基线部署，不要一开始就做 `INT8`。

验证内容包括：

- 模型是否能成功转成 engine
- 是否有 unsupported operator
- 推理输出是否与训练侧结果基本对齐
- 延迟是否满足预期
- 显存占用是否可接受

如果遇到 unsupported operator，需要立即做决策：

- 能否改图替换成标准算子
- 能否把该模块拆出去
- 是否值得为该算子编写 plugin

### 6.5 第五步：补齐 plugin 或 CUDA 模块

对于 TensorRT 不支持但又无法绕开的关键算子，可以考虑：

- 自定义 TensorRT plugin
- 自定义 CUDA kernel
- 单独构建运行模块，再与 TensorRT engine 级联

这一阶段要控制范围，优先解决“核心路径上的少量关键算子”，避免把整个工程变成自定义插件集合。

### 6.6 第六步：在 Orin 真机上做性能测试

必须在目标 Orin 环境上做测试，不建议仅在 PC 侧验证。

重点测试内容：

- 单帧延迟
- 平均吞吐
- 峰值显存
- 长时间运行稳定性
- 温升和降频情况
- 多传感器输入下的抖动

同时建议固定功耗和频率状态，保证数据可比。

### 6.7 第七步：完成整车接入

整车接入时还需要处理：

- 相机或点云数据接入
- 时间同步
- 多线程调度
- CUDA stream 管理
- 内存拷贝优化
- 模块 watchdog
- 异常恢复机制
- 结果发布接口

真正的车端部署难点，通常不只在“模型能跑”，更在于“系统长期稳定运行”。

## 7. camera-only OCC 与 lidar OCC 的差异

### 7.1 camera-only OCC

如果项目以多相机为主，通常更推荐这条路线，因为它更容易做成规则的 dense 网络：

- 图像 backbone 容易转 TensorRT
- BEV 特征表达更规整
- 部署链条更贴近现有 NVIDIA 推理体系

适合做法：

- 多相机输入固定分辨率
- Backbone 和 Neck 走 TensorRT
- View Transform 根据复杂度决定是否单独实现
- OCC Head 继续走 TensorRT

### 7.2 lidar OCC

如果是 lidar OCC，部署难点通常集中在：

- voxelization
- sparse conv
- scatter/gather
- 稀疏张量表达

如果该方案高度依赖 `spconv`，在 Orin 上往往不够友好。建议优先考虑：

- `pillar` 路线
- `range view` 路线
- 尽早转换到 `dense BEV`

如果必须保留稀疏卷积，则需要接受以下工程现实：

- 需要 plugin 或自定义 CUDA
- TensorRT 主流程无法全自动处理
- 排障成本会明显升高

## 8. 工程建议

### 8.1 第一阶段目标

第一阶段不要追求极限性能，而要追求：

- 正确跑通
- 输出稳定
- 数值可对齐
- 延迟可测
- 系统不崩

### 8.2 第二阶段目标

在 FP16 跑通之后，再考虑：

- 图优化
- engine 融合
- plugin 优化
- 内存复用
- CUDA graph
- 多 stream 并行
- `INT8` 校准

### 8.3 不建议的做法

- 训练代码未经裁剪直接上车
- 未固定版本就开始写 plugin
- 模型还未收敛就提前深度部署优化
- 复杂算子过多时仍坚持整图单 engine
- 没有真机压测就进入整车联调

---

## 9. 一份可执行的落地建议

如果当前要尽快推进 Orin 上的 OCC 部署，推荐按如下顺序执行：

1. 确认最终要部署的 OCC 模型
2. 梳理模型算子和模块依赖
3. 判断属于哪一类模型：
   - `A 类`：dense BEV 主导，部署友好
   - `B 类`：含大量 sparse/custom 算子，部署困难
4. 先导出最小 ONNX 子图
5. 在 Orin 上做 FP16 TensorRT 基线
6. 收集 unsupported operator 清单
7. 对每个问题决定是改图、拆模块还是写 plugin
8. 完成端到端链路打通
9. 再做性能优化和稳定性测试

---

## 10. 当前最推荐的判断标准

如果你现在手里的 OCC 网络满足以下特征，则比较适合优先在 Orin 上落地：

- 主体是规则 dense 网络
- 算子大部分能转 ONNX
- 不强依赖稀疏卷积
- 输入 shape 可固定
- batch size 可固定为 `1`

如果网络具有以下特征，则要提前预估插件开发工作量：

- 依赖 `spconv`
- 大量稀疏体素算子
- 自定义采样或 pooling
- 非标准 attention
- 动态 shape 很多

---

## 11. 建议的下一步

基于当前信息，建议下一步不要继续泛泛调研，而是进入“模型体检”阶段。优先准备以下材料：

- 模型名称或论文名称
- 网络结构图
- ONNX 导出结果
- TensorRT 转换日志
- 关键算子列表

拿到这些信息之后，就可以进一步输出：

- 哪些模块能直接上 TensorRT
- 哪些模块需要改图
- 哪些模块需要 plugin
- 最终应该拆成几个推理模块

---

## 12. 总结

在 Orin 上部署 OCC 是可行的，但要用车端部署思维来做，而不是训练侧思维。

最推荐的主线是：

- 选择部署友好的 OCC 架构
- 优先走 `GPU + TensorRT + FP16`
- 尽量采用 `dense BEV` 路线
- 将难部署模块拆开处理
- 在 Orin 真机上构建和测试

如果模型高度依赖稀疏卷积或复杂自定义算子，那么这项工作会迅速演变为“插件开发 + 系统优化”项目，周期和风险都要提前预留。

从工程可控性来看，优先选择部署友好的模型结构，是在 Orin 上落地 OCC 的关键。
